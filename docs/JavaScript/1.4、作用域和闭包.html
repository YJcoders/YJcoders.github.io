    作用域和闭包

    YJcoders 10/13/2019 JS

    # 1、作用域

    作用域就是变量和函数可访问的范围；

1、函数作用域：
// 函数内部声明变量
function f() { var a = 10;  return a }
f()
console.log(a) // 报错： a is not defined
// 没有使用var关键字，声明全局变量，（不建议这样使用）
function f() { a = 10;  return a }
f()
console.log(a) // 10
2、全局作用域：
var b = 'bar'
function foo() {
  console.log(b)
}
foo() // 全局变量 b

3、作用域链：沿着作用域一层层向外寻找变量，直到找到这个过程，称之为作用域链
var b = 'bar'
function bar() {
  function foo() {
      console.log(b)
  }
  foo()
}
bar() // bar

4、es6 块级作用域和暂时性死区
let, const , 变量声明关键字; 和 {}
function foo() {
  console.log(bar)
  var bar = 3
}
foo() // undefined
// 变量提升解析
function foo() {
  var bar;
  console.log(bar);
  bar = 3
}
foo() // undefined

let 或 const 声明变量，会针对这个变量形成一个封闭的块级作用域，在这个块级作用域当中，
如果在声明变量前访问该变量，就会报 referenceError 错误
function foo() {
  // console.log(bar)
  console.log(bar1)
  let bar = 3
  const bar1 = 2
}
foo() // 报错 ReferenceError

    # 2、执行上下文和调用栈

    执行上下文就是当前代码的执行环境/作用域；分为全局上下文和函数上下文
执行上下文特点 ：
1、单线程，主进程执行
2、同步执行，按顺序执行

全局执行上下文中：
1. 创建一个全局对象，在浏览器中这个全局对象就是 window 对象。
2. 将 this 指针指向这个全局对象。一个程序中只能存在一个全局执行上下文

函数执行上下文：
每次调用函数时，都会为该函数创建一个新的执行上下文。
每个函数都拥有自己的执行上下文，但是只有在函数被调用的时候才会被创建。
一个程序中可以存在任意数量的函数执行上下文。


3、全局上下文只有一个，浏览器关闭时会被弹出栈
4、函数上下文没有数量限制，每次执行函数就会生成一个函数上下文，执行完后出栈，等待被浏览器回收销毁

js执行分为两个阶段：代码预创建阶段；代码执行阶段

创建阶段，变量，函数提升(函数声明优先级高于变量声明),确定作用域链，确定this指向；执行阶段，变量,函数赋值，代码执行
var bar = function () {
  console.log('bar2')
}
function bar() {
  console.log('bar1')
}
bar() // bar2
// 变量提升解析
function bar() {
  console.log('bar1')
}
var bar;
bar = function () {
  console.log('bar2')
}
bar() // bar2

// 调用栈
function foo1() {
  foo2()
}
function foo2() {
  foo3()
}
function foo3() {
  foo4()
}
function foo4() {
  console.log('foo4')
}
foo1()

调用关系：foo1 → foo2 → foo3 → foo4。这个过程是 foo1 先入栈，紧接着 foo1 调用 foo2，foo2入栈，以此类推，foo3、foo4，
直到 foo4 执行完 —— foo4 先出栈，foo3 再出栈，接着是 foo2 出栈，最后是 foo1 出栈。这个过程“先进后出”（“后进先出”）

    # 3、闭包

    闭包是指有权访问另一个函数作用域中变量的函数
函数嵌套函数时，内部函数使用到外部函数作用域下的的变量，并且内部函数在全局作用域下可以访问，就形成了闭包
函数内部可以读取函数外部的变量，但是函数外部不能访问函数内部的变量

闭包用途：
1、定义私有变量，避免变量被污染
2、让变量始终保持在内存中，不被回收
3、防抖、节流
4、模块化开发
5、异步操作中的回调函数

使用闭包的注意点：
由于闭包会使得函数中的变量都被保存在内存中，内存消耗很大，所以不能滥用闭包，否则会造成网页的性能问题，在IE中可能导致内存泄露。
解决方法是，在退出函数之前，将不使用的局部变量全部删除。

    # 4、垃圾回收机制

    js本身具有垃圾回收机制，执行环境（浏览器）会负责代码执行过程的内存管理

注意：　正常来讲，在函数执行完毕并出栈时，函数内局部变量在下一个垃圾回收节点会被回收，该函数对应的执行上下文将会被销毁，这也正是我们在外界无法访问函数内定义的变量的原因。也就是说，只有在函数执行时，相关函数可以访问该变量，该变量在预编译阶段进行创建，在执行阶段进行激活，在函数执行完毕后，相关上下文被销毁

垃圾回收一般有两种清除策略：1、标记清除；2、引用计数
所以在变量使用完之后一般要清除
var foo = 'bar' // 在堆内存中给变量分配空间
alert(foo)  // 使用内存
foo = null // 释放内存空间

栈空间：由操作系统自动分配释放，存放函数的参数值，局部变量的值等
堆空间：一般由开发者分配释放，这部分空间就要考虑垃圾回收的问题。

    # 5、内存泄漏

    内存泄漏是指内存空间明明已经不再被使用，但由于某种原因并没有被释放的现象
// 常见的内存泄漏
1、意外的全局变量
function foo(arg) {
    bar = "this is a hidden global variable";
}
2、脱离 DOM 的引用
3、闭包

// 例1，使用完清除变量
var element = document.getElementById("element")
console.log(element)
element.mark = "marked"

// 移除 element 节点
function remove() {
    element.parentNode.removeChild(element)
    // element = null // 清理内存
}
remove()
console.log(element, 990) // dom中已经移除element, 可是此处还是可以访问
// 解决办法就是在remove方法中添加  element = null

// 例二，事件监听，没必要一直存在就清除
var element = document.getElementById('element')
element.innerHTML = '<button id="button">点击</button>'

// 事件监听一直存在，占用内存
var button = document.getElementById('button')
button.addEventListener('click', function() {
    // ...
})

element.innerHTML = '' // dom中已经移除,但是window内存里面还有button
console.log(button, 999) // 解决方法就是取消事件监听

// 例三，定时器 setTimeout, setInterval  也要及时清除
function foo() {
    var name  = 'lucas'
    window.setInterval(function() {
        console.log(name)
    }, 1000)
}
foo()
// 会一直打印name值
